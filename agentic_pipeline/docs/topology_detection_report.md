# OSM-based topology detection report

Generated by running `osm_enrichment.detect_topology()` against all 19
reports in `manual_classification_reference.md`. This automates what was
previously a manual Google Maps/OSM lookup ("is this a straight road or a
4-way intersection?") for selecting between `straight_road.xodr` and
`intersection_4way.xodr`.

Method: street name(s) are extracted deterministically from the report text
(regex-based, not LLM — the *decision* must be deterministic; see
`_extract_street_candidates()`'s docstring in `osm_enrichment.py`). With
exactly one street name, the location is `midblock`. With exactly two, the
existing `_intersection_heading()`-style Overpass lookup finds the shared
OSM node between them, and the number of distinct ways touching that node
decides the topology: 2 ways → not actually a junction → `midblock`; 4 ways
→ `4way_junction`; anything else (3, 5+, no shared node, failed geocode) →
`needs_manual_review` — no template is forced.

## needs_manual_review — reason for each

- **turning_08** (Reinickendorfer Straße / Pankstraße): shared node found,
  but only **3** ways touch it, not the clean 2-or-4 case.
- **turning_09** (Straße zum Müggelhort / Müggelheimer Damm): shared node
  found, **3** ways at it.
- **crossing_02** (Rathausstraße / Poststraße / Spreeufer): **3 street names**
  extracted from the text (the report describes the cyclist's route as
  "from Poststraße, on Rathausstraße, toward Spreeufer" — no single named
  cross-street defines the collision site). The decision logic is only
  defined for exactly 1 or 2 street names; picking an arbitrary 2 of 3
  would be a guess, so this is left for manual review.
- **crossing_03** (Müggelheimer Damm / Waldnesselweg / Erwin-Bock Str.):
  same — 3 street names, an explicitly named 3-street corridor ("Kreuzung
  Waldnesselweg/Erwin-Bock Str." crossing Müggelheimer Damm"). Genuinely
  ambiguous which 2 to query without guessing.
- **crossing_06** (Oranienburger Straße / Taldorfer Weg): exactly 2 streets
  extracted, but no shared OSM node was found within 5km of the geocoded
  location. Checked manually: "Taldorfer Weg" geocodes to Wittenau
  (Reinickendorf, far NW Berlin) while "Oranienburger Straße" geocodes to
  Mitte — the same kind of wrong-instance-of-an-ambiguous-name problem as
  turning_06/schoenhauser, just not yet manually verified/overridden.
- **crossing_08** (Unter den Eichen / Drakestraße / Habelschwerdter Allee):
  **3 street names** — the report *explicitly* names a 3-way intersection
  ("Kreuzungsbereich Unter den Eichen / Drakestraße / Habelschwerdter
  Allee"). Extraction is correct here; the ambiguity is real (which 2 of
  the 3 to query), so this is a case where "3 candidates" reflects genuine
  complexity, not an extraction error.
- **longitudinal_01** (Lötschbergstraße / Braunsdorfstraße): no shared node
  found. Both names geocode to the same neighborhood (Alt-Biesdorf), but
  they are likely two different landmark points *along* the primary road
  ("Straße Alt-Biesdorf", which the extractor missed — "Straße X" as a
  bare leading noun before a place name isn't one of the patterns it
  handles), not real cross-streets. This is a lane-change/longitudinal
  report, which doesn't actually need topology to pick its template (see
  Wiring, below) — flagged for completeness, not urgency.
- **longitudinal_02** (Markgrafendamm / Hauptstraße / Persiusstraße): **3
  street names** — same "not exactly 2" case; also a lane-change report.

## turning_06 (schoenhauser) — manual override, not a fresh finding

This report already has a manually-verified coordinate override from this
session's earlier heading fix (commits `d6431cd`/`96adbf4`) — its "Schönhauser
Straße" wrongly geocodes ~7km away in Pankow. `detect_topology()` reuses that
same verified coordinate (52.528644, 13.409324) via a parallel
`_manual_topology_override()` (same pattern, not duplicated logic). Checking
the actual shared node there (this session) found 4 ways touching it —
Schönhauser Allee, Alte Schönhauser Straße, and Torstraße split into two
segments — a genuine 4-way crossing, not a forced/guessed answer.

## Disagreements between topology and scenario_type — flagged, not resolved

Several reports land in a topology that a naive reading of their
scenario_type/md-heading would not predict. These are reported as **data
quality signals**, not silently corrected:

- **turning_02** and **turning_07** are under the md's `## TURNING` heading
  (real turn maneuvers happened) but land as `midblock` — because the report
  text only names **one** street. For turning_02 (LKW turns right into
  Mollstraße) this is a genuine gap: a real turn happened, but the road the
  truck turned *from* is never named, so there's no second street for the
  topology check to use. For turning_07 (LKW turns into a parking lot) this
  is arguably *correct*, not a disagreement — turning into a parking lot
  isn't a road junction at all.
- **crossing_01** (Mühlenstr., cyclist enters the roadway from the sidewalk
  mid-block) and **crossing_04** (Landsberger Allee, cyclist crosses at a
  median crossing aid) are under the md's `## CROSSING` heading but land as
  `midblock`. This matches an observation already flagged in this session's
  generate_scenario.py work: several "CROSSING"-labeled reports describe
  midblock/enter-roadway incidents, not intersection crossings — the md's
  category groups by *maneuver*, not by physical topology, and this is
  exactly the kind of case where the two diverge.
- **crossing_05** (Storkower Straße) also lands as `midblock` — this report
  actually describes a *cyclist* turning left (not the car), with no
  cross-street named at all; another instance of the maneuver category not
  matching the physical topology.

None of these were "fixed" by picking a different template than what
`detect_topology()` found — they're reported as-is so a human can judge
whether the underlying classification or the report's phrasing needs
attention.

## Wiring — where this plugs into the pipeline

- **`template_selector.py`**: `select_template()` now takes an optional
  `topology` argument (the `detect_topology()` result) and uses it as the
  deciding input for `turning` / `crossing` / `other`, replacing the previous
  scenario_type-only lookup table for those types. (Note: this document was
  originally written against the earlier 9-category scenario_type system —
  right_turn_conflict/left_turn_conflict/straight_crossing_conflict/etc.
  Classification has since been rebuilt around a 4-category taxonomy —
  turning/crossing/longitudinal/other — but the topology results above, which
  never depended on scenario_type at all, are unaffected.) `longitudinal`
  still always gets `straight_road.xodr` regardless of topology — a
  longitudinal (overtaking/lane-change) scenario is a single-road maneuver by
  definition, and the street names these reports mention are incidental
  landmarks along that road, not a real cross-street (see
  longitudinal_01/02 above). When topology is `None` or
  `needs_manual_review`, the scenario_type-based table is used as a
  fallback so generation isn't blocked while a report awaits manual review.
- **`pipeline.py`**: `_tool_generate_scenario()` now calls
  `detect_topology(report_text, scenario_id)` (reusing
  `data["source"]["raw_text"]`, already present from extraction — no new
  state needed) right before selecting the template, records the result via
  `state.record("detect_topology", ...)`, stores it at `state.data["topology"]`
  and the resolved filename at `state.data["template_used"]` for
  traceability (both end up in the `.enriched.json` output), and prints a
  topology summary line.
- **`run_all.py`**: the `[r]/[f]/[ok]` review loop now prints a line showing
  `topology`, `streets`, `way_count`, and the resolved `template` alongside
  the existing `scenario_type`/`confidence` line.
- **`generate_road.py`**: **no changes** — it is not imported or called
  anywhere in the active pipeline (`pipeline.py` copies a pre-validated
  static `.xodr` template via `shutil.copy2`, it does not call
  `generate_opendrive()`). It appears to be separate, not-yet-wired-in work.
- **`generate_scenario.py`**: **no changes** — its junction-vs-straight
  branch (`_is_junction_template()`, added for Bug 2) already keys off the
  *actual resolved template filename*, not scenario_type, so it
  transparently honors whatever `select_template()` decides now, with or
  without a topology signal. Not touched, per this task's scope.
